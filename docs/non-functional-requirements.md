# 非機能要件（v0最低ライン）

## 前提条件

このドキュメント全体は以下の前提で書かれている。後続の性能・可用性・運用要件はこの前提に依存する。

- **組織形態**: 会社運営（個人運用ではない）
- **想定ユーザー**: 社内スタッフ2〜3人。全員同権限・ロール分けなし
- **対象施設**: 1施設（v0段階）
- **アプリ形態**: 独立Railsアプリ（landbase_ai_suite との同居なし）

> v0着手前に決めておかないと、後から変更すると過去データに遡及できない or 致命傷になる項目だけを絞って明文化する。
> 本格的な非機能要件は v1以降に拡張する。

---

## 0. 方針

- 上記前提のもと、**税務データ**を扱うため、保存期間・バックアップ・改ざん耐性の3点は v0でも妥協しない
- 認証・最低限の操作履歴は v0段階から必要（複数スタッフが触るため）
- 性能・本格的な監査ログ・SSO・多要素認証等は v0では割り切る

---

## 1. データ保存期間 ⚠️必須

### 要件

- **すべての宿泊実績・領収書・月次集計データを 7年間 保持する**

### 根拠

| 法令 | 保存期間 |
|---|---|
| 国税関係帳簿（所得税法・法人税法） | 7年（青色申告で欠損金がある場合10年） |
| 消費税法 | 7年 |
| 電子帳簿保存法（電子保存する場合） | 7年 |
| 住宅宿泊事業法（宿泊者名簿） | 3年 |

→ 最長を取って **「7年」を v0の業務上の標準** とする。
→ ただし、将来「欠損金がある事業年度」になった場合に備え、**システム上は最大10年まで保持期間を伸ばせる構造**にしておく。
→ つまり「**業務上は7年で十分だが、システム上は最大10年まで伸ばせる**」という二段構え。
   運用ポリシーは7年からスタートし、必要に応じて10年に切り替えられる。

### 設計への影響

- DB は7年分のデータを安価に保持できるものを選ぶ（技術スタック選定の判断材料）
- 「アーカイブテーブルへの移動」のような複雑化はv0では避け、単一テーブルで素直に持つ
- バックアップも7年分をどこかで保持する必要がある（後述）

---

## 2. データ消失耐性・DR ⚠️必須

### RPO / RTO

| 指標 | 目標値 | 補足 |
|---|---|---|
| **RPO**（許容データ損失） | **24時間以内** | 毎日1回バックアップを取る前提。最悪1日分のデータ再入力で済む |
| **RTO**（復旧目標時間） | **1時間以内** | 障害検知から本番稼働まで。月末月初の申告期はこれを死守 |
| 復元手順の起動時間 | **5分以内** | 手順書を見ながら復元コマンドを開始するまでの時間（RTOとは別指標） |

### 要件

- 自動バックアップを **毎日1回以上** 実行する
- バックアップは **本番DBとは物理的に別の場所** に保存する
  - **最低条件**: 同一リージョン内の **別AZ（アベイラビリティゾーン）** に複製
  - **推奨**: **別リージョン** にも世代バックアップを置く（自然災害・リージョン障害対策）
  - 同じディスク・同じVMインスタンスに置くのはNG
- バックアップからの **復元手順は手順書化** し、5分以内にコマンドを起動できる状態にする
- バックアップの保持期間は **30日以上**（誤削除・誤更新に気づくまでの猶予）

### 設計への影響

- マネージドDB（自動バックアップ機能を持つもの）を強く推奨
- セルフホストSQLiteの場合は外部ストレージ（S3互換ストレージ等）への定期スナップショット必須
- 復元手順は `docs/runbook/restore.md` として残す（実装フェーズのタスク）

---

## 3. 改ざん耐性・電帳法対応の最低ライン

> **法対応の解釈レベル**: v0時点では **顧問税理士の正式レビューを前提としない簡易対応** に留める。
> 「将来、顧問税理士のチェックを受けたときに過去データが対象外にならない」ための最低ラインのみを満たす。
> 本格的な電帳法準拠（JIIMA認証相当）は v1以降に専門家レビューを受けて対応する。

### 要件（v0で守る）

- **領収書PDFは生成後に上書き禁止**。再発行する場合は新しいファイルを生成し、元PDFも保持する
- **宿泊実績の修正は履歴を残す**。最低限「いつ・誰が・どのフィールドを変更したか」を記録できる構造にする
  - 複数スタッフが触るため、`updated_by_user_id` を必須化
- **検索要件**: 日付・金額・宛名（取引先）で検索できる
- **取消は物理削除しない**。`status = cancelled` にして残す

### v0でやらないこと

- ❌ タイムスタンプ局によるタイムスタンプ付与
- ❌ 訂正・削除の不可逆ログ署名
- ❌ JIIMA認証取得
- ❌ スキャナ保存制度対応
- ❌ 真実性確保のための専用システム要件

→ 本格的な電帳法対応は v1以降。v0時点で「将来対応するときに過去データが対象外にならない」最低ラインだけ守る。

---

## 4. セキュリティ・認証

### 要件（v0最低ライン）

- **複数ユーザーアカウント制を v0段階から実装**
  - スタッフ2〜3人がそれぞれ自分のアカウントでログインする
  - パスワード認証（または Magic Link）必須。無認証は禁止
  - 全員同権限（ロール分けはなし）
  - ユーザーの追加・無効化は管理画面 or 設定で可能（コードを書き換えなくて良いこと）
- 本番URLは公開ディレクトリ（検索エンジン）に載せない（noindex / robots.txt / 公開しない）
- DBへの直接アクセスは管理者のみ
- 領収書PDF・エクスポートCSV は **認証越しでないとダウンロードできない**（URL直叩き禁止 = 推測不能なIDでも認証必須）
- 通信は HTTPS のみ
- 個人情報（宿泊者氏名等）を含むデータは **at-rest 暗号化** されているサービス・ストレージに置く

### 管理操作の監査（最低ライン）

v0では「ロール分けなし・全員同権限」だが、税務データを扱う以上、最低限の監査観点は持つ:

- **ユーザー追加・無効化** の操作は履歴に残す（誰が・いつ・誰を追加/無効化したか）
- **月次確定** の操作は履歴に残す（誰が・いつ・どの月を確定したか）
- **領収書発行** の操作は履歴に残す（誰が・いつ・何番の領収書を発行したか）
- 上記の操作ログは **最低2年保持**（本データの7年保持とは別建てでよい）

### v0でやらないこと

- ❌ 多要素認証
- ❌ SSO（landbase_ai_suite との SSO 連携は v1以降に検討）
- ❌ ロールベース権限
- ❌ IPホワイトリスト
- ❌ WAF
- ❌ ペネトレーションテスト

## 5. 性能・可用性

### 想定規模

- 月の宿泊レコード: **数十〜数百件**
- 利用ユーザー: **2〜3人（社内スタッフ）**
- 平常時の同時アクセス: **1〜2人**（朝・夕に1人ずつ入力する程度）
- ピーク同時アクセス: **多くて3人**（月次確定時にスタッフが集まる程度）
- 7年で総レコード: 数万件オーダー

### 申告締め切り期間（高負荷期）

- **想定**: 翌月の月初〜中旬に前月分の申告作業を集中して行う（沖縄県条例の最終確定により調整、要 Open Questions）
- この期間は次節「強化監視」を適用する

### 要件

- **性能要件は事実上なし**（上記規模ならどんな構成でも問題なく動く）
  - 参考目標: 平常時 p95 レスポンスタイム < 1s、同時アクセス3人で破綻しないこと
- 可用性目標: **99%**（年間 約3.6日 までの停止許容）
- ただし **申告締め切り期間** は次の「強化監視」を適用する
- 停止しても税額や過去データが消えるわけではないので、SLA契約等は不要

### 強化監視（申告締め切り期間のみ）

平常時は最低限の死活監視だけ。申告期間は次の閾値で**メールor Slack 通知**を仕込む:

- **エラーレート > 1%** が5分以上続いたらアラート
- **p95 レスポンスタイム > 2秒** が5分以上続いたらアラート
- **5xx 系エラー > 5件/分** でアラート
- **DB接続失敗 > 3回/分** でアラート

通知の宛先・方法（メール/Slack/etc）は実装フェーズで決定する。

---

## 6. 個人情報

- 宿泊者の代表者氏名・連絡先は個人情報として扱う
- DB暗号化（at-rest）必須（マネージドDBの標準機能で達成可）
- バックアップにも暗号化が及ぶこと
- 個人情報の取得・利用・保管目的を明示するプライバシーポリシーを v1までに用意（v0では社内運用のため後回し可）

---

## 7. 監査ログ

### v0要件

- 各レコードに `created_by` / `updated_by`（ユーザーID）と `created_at` / `updated_at` を持つ
  - 複数スタッフが触るため「誰が変更したか」は v0段階から最低限必要
- 修正・取消は物理削除せず履歴扱い（§3参照）
- 修正前の値を保持する仕組みは v0で持つ（papertrail 等のシンプルな履歴管理 gem を想定）
- 専用の高機能な audit_log テーブル（操作ログ全種類を別テーブルに記録）は v0では作らない

### v1以降に倒すこと

- ログイン履歴の本格管理
- 操作ログの全種類記録（閲覧含む）
- ログ改ざん防止（タイムスタンプ局・ハッシュチェイン等）

---

## 8. 環境・リリース

- 環境は **本番1つだけ**（v0では開発環境=ローカルでよい、ステージング不要）
- リリースは手動デプロイで十分
- ロールバック手順は最低限スクリプト化しておく

## 9. 技術スタック選定への影響まとめ

このドキュメントから次工程（技術スタック選定）に渡す制約:

| カテゴリ | 制約 |
|---|---|
| **DB** | 7年分（数万レコード）を安価に保持できる / 自動バックアップ機能あり / at-rest暗号化標準 |
| **バックアップ** | 毎日自動 / 別AZ以上に複製 / 30日保持 / RPO 24h・RTO 1h を満たせる |
| **ホスティング** | HTTPS標準 / 認証越しのファイル配信ができる |
| **PDF生成** | 生成済みファイルを永続保存できる / 上書きしない運用が可能 |
| **認証** | 複数ユーザーアカウント制（2〜3人）。Devise / Rodauth 等で実装容易 |
| **履歴管理** | レコード変更履歴を持てる仕組み（papertrail 等が利用可能） |
| **構成** | 独立Railsアプリとして動作（landbase_ai_suite との同居はしない） |
| **監視** | 申告期間中の最低限のアラート（エラーレート・p95・5xx）が仕込める |
| **コスト感** | 月額数百〜数千円レベルで上記を満たせること |

### 候補スタック想定時の重点ポイント（読み手向けメモ）

技術スタック選定は **Ruby on Rails + PostgreSQL** を第一候補として進める前提。
このスタックを想定したとき、特に効いてくる制約は次の3つ:

1. **PostgreSQL のマネージドサービスを選ぶ理由が強い** — §1（7年保持）+ §2（自動バックアップ・別AZ）+ §6（at-rest暗号化）を満たす最短ルート
   - 例: AWS RDS / Google Cloud SQL / Render Postgres / Fly.io Postgres / Supabase 等
2. **オブジェクトストレージが必須** — §3 の領収書PDF永続保存・上書き禁止運用は、ローカルディスクではなく S3互換ストレージ（AWS S3 / R2 / GCS 等）で実現するのが自然
3. **デプロイ先は HTTPS・認証必須を標準で満たすPaaS** が望ましい — §4 のセキュリティ要件を自前で組まずに済ませるため
   - 例: Render / Fly.io / Heroku / AWS App Runner 等

これらは「絶対」ではなく、技術スタック選定時に **トレードオフを意識して読むべきポイント**として残しておく。
