# 紙台帳テンプレート（v0）

> v0は「紙台帳に手書き → スタッフが画面から転記入力 → 税計算 → 月次集計」の一筆書き運用を前提とする。
> 紙台帳とシステムの Stay モデルを **1:1 対応**で設計することで、転記時の解釈ブレ・カラム漏れを防ぐ。

## 1. 設計方針

### 1.1 1予約 = 1行

[v0要件 §2 入力単位の方針](./v0-requirements.md#入力単位の方針) に従い、紙台帳でも **1予約 = 1行**で記録する。

- 連泊は「泊数」欄に数字を書き、行を増やさない
- 1予約で複数部屋取るケースは v0 では想定しない（民泊1棟貸し前提）。発生したら2行に分割する

### 1.2 Stay モデルとの 1:1 対応

紙台帳の各列は、システム側 [Stay モデル](./v0-requirements.md#stay宿泊実績) のカラムと **1:1 対応**する。
転記時に「この欄はどのカラムか」で迷わないことを最優先する。

> **注**: Stay モデルには紙台帳に列を持たないメタデータカラム（`payment_method`、`created_by_user_id`、`updated_by_user_id`、`tax_rule_version` 等）も存在する。これらは紙台帳からの転記対象ではなく、システム側で自動記録 or iPad 現場入力運用時に入力する。紙台帳と 1:1 対応するのは **現場で人間が記入する業務情報カラム** に限定される。

### 1.3 計算欄は紙では書かない

`taxable_amount` / `tax_amount` は**システム側で自動計算**するため、紙台帳では空欄または「計算欄」マークのみ。
スタッフが暗算する必要はない（入力ミスの温床になるため）。

### 1.4 宿泊者名簿との分離

住宅宿泊事業法の **宿泊者名簿**（宿泊者全員の氏名・住所・職業・国籍・旅券番号等）は、本紙台帳とは**別の様式**で管理する。
本紙台帳は「税計算と月次集計に必要な最小情報」のみを持つ。

> 宿泊者名簿の自動生成・提出機能は v1以降のスコープ（[v0要件 §2 v0でやらないこと](./v0-requirements.md#v0でやらないこと明示) 参照）。

## 2. 紙台帳の列定義

A4横で印刷することを想定した列構成（左から順）。

| # | 列名 | 対応する Stay カラム | 必須 | 記入例 | 備考 |
|---|---|---|---|---|---|
| 1 | 受付№ | (なし。紙台帳通番) |  | 2026-04-001 | 月初リセットで連番 |
| 2 | チェックイン日 | `check_in_date` | ✓ | 4/1 | 月単位で台帳を分けるなら日付のみ可 |
| 3 | 泊数 | `nights` | ✓ | 2 | 連泊数 |
| 4 | チェックアウト日 | (導出: `check_in_date + nights`) |  | 4/3 | 確認用。記入は任意 |
| 5 | 代表者氏名 | `guest_name` | ✓ | 山田 太郎 |  |
| 6 | 総人数 | `num_guests` | ✓ | 3 |  |
| 7 | 課税対象人数 | `num_taxable_guests` | ✓ | 3 | 通常は総人数と同じ |
| 8 | 免除人数 | `num_exempt_guests` | ✓ | 0 | 通常は 0 |
| 9 | 免除理由 | `exemption_reason` |  | (空欄) | 修学旅行等の場合のみ記入 |
| 10 | 1人1泊料金（税抜） | `nightly_rate` | ✓ | 8,000 | 円。税抜 |
| 11 | 課税対象合計 | `taxable_amount` |  | （計算欄） | システム自動計算 |
| 12 | 宿泊税額 | `tax_amount` |  | （計算欄） | システム自動計算 |
| 13 | チャネル | `channel` |  | Airbnb | プリセットから選択 |
| 14 | 予約番号 | `external_reservation_id` |  | HM-XYZ123 | OTA側の予約ID |
| 15 | 備考 | `memo` |  |  | 自由記入 |
| 16 | 入力済 | (なし。運用フラグ) |  | ✓ | システム転記後にチェック |

### 2.1 列定義の補足

- **#1 受付№**: システム側にはないが、紙と画面の照合用に紙独自で持つ。`yyyy-mm-連番` で運用
- **#4 チェックアウト日**: システム側はチェックイン日 + 泊数で導出するが、**人間の確認用に紙には書く**（書き間違い検出のため）
- **#7-#9 課税/免除人数**: 整合チェック `num_guests = num_taxable_guests + num_exempt_guests` を満たすよう注意
- **#10 1人1泊料金**: **税抜**で記入する。税込みで入力すると税額が二重計上になる
- **#11-#12 計算欄**: 紙では空欄にしておき、システム入力後の控えとして印刷時に書き戻す運用も可
- **#13 チャネル**: フリーテキストではなく、`config/facility.json` の `channels` で定義したプリセット名を使う（揺らぎ防止）
- **#16 入力済**: スタッフが画面入力を完了したらチェックを入れる。**未入力の取りこぼしを防ぐ運用フラグ**

### 3.1 印刷物の体裁

- A4横、上下マージン15mm
- 1ページあたり 20行程度（民泊1棟貸しなら月20件前後で十分）
- 月ごとに別ファイル/別綴りで保管
- 表紙に「○○ゲストハウス 宿泊台帳 2026年4月分」等を記載

### 3.2 印刷可能な PDF 版

実際の印刷物（PDFテンプレート）は v0実装時に Prawn で生成する。
本ドキュメントは「列定義と運用ルールの正」とし、PDF はそこから派生させる。

## 4. 運用フロー

### 4.1 日次

1. チェックイン受付時、その場で紙台帳に記入する
2. 受付№を採番し、IN/泊数/氏名/人数/料金/チャネル/予約番号を埋める
3. 課税/免除人数は通常「総人数 = 課税対象、免除 0」でよい
4. 修学旅行等の課税免除がある場合のみ免除人数と理由を書く

### 4.2 翌営業日（または週次）

1. スタッフが紙台帳を見ながら画面から転記入力
2. システムが税額を自動計算 → 紙の計算欄に書き戻す（必要なら）
3. 入力完了したら **「入力済」欄にチェック**を入れる
4. 未チェック行が残っていれば翌日へ持ち越し

### 4.3 月次

1. 月末に「入力済」欄が全行チェックされているか確認
2. システムの月次集計画面で紙台帳の合計と突き合わせる
3. 一致したら「月次確定」ボタンで対象月を編集ロック
4. 紙台帳は月次ファイルに綴じて保管

## 5. 保管・廃棄

- **保管期間**: [非機能要件 §3](./non-functional-requirements.md) に揃え、**業務上7年・システム上10年**を満たすよう物理保管する
- **保管場所**: 鍵付きキャビネット（個人情報を含むため）
- **廃棄**: 保管期間経過後、シュレッダー処理
- **電子化**: スキャンしてPDF保存する場合も、原本は保管期間中は破棄しない

## 6. 既知の限界・将来の拡張

### 6.1 v0の限界

- 紙運用のため、転記の手間と転記ミスのリスクがある
- 「入力済」チェックの運用が崩れると取りこぼしが発生する
- 税額を紙に書き戻す運用は手間なので、実運用次第で省略可

### 6.2 v1以降の改善案

- **OTA連携 (iCal取込)**: Airbnb 等の予約を自動で Stay レコード化し、紙台帳の役割を「現場確認・サイン用」に縮退
- **タブレット入力**: 受付時にその場で画面入力し、紙台帳を完全に置き換える
- **宿泊者名簿の自動生成**: 紙台帳とは別管理だった宿泊者名簿もシステム側で生成
