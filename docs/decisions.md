# 意思決定ログ

## このファイルについて

チーム検討段階における設計・方針判断の「なぜその結論に至ったか」を時系列で記録するファイル。チャットログや Issue コメントは消失・散逸しやすいため、残すべき方針レベルの判断はここに集約する。将来メンバーが加わった際に、決定理由を再構築するコストを下げることが目的。

## 書き方

- 1判断あたり10〜30行目安、フォーマットは自由
- 各項目は **決定 / 背景 / 代替案 / 影響** の4点をゆるく守る
- 完璧を目指さない。書き漏れは後から追記でよい
- 決定が覆った場合は、元の記述を消さず「**覆された**」と追記し、新しい判断を別項目として追加する
- 実装細部や作業手順は各 Issue/PR に任せる。ここに記録するのは方針レベルの判断のみ

## 2026-04-08: Mac / iPad / Square 前提デバイスと助成金キット販売モデル

**決定**: アプリの動作要件は「モダンブラウザ（Chrome / Safari / Edge 最新版）が動作する任意の PC」で OS 非依存を維持する（**推奨は Mac** — 販売キットで同梱するため動作検証・サポート窓口を集約したい）。加えて **iPad をオプションの現場入力デバイスとして v0 から対応**し、**Square を決済端末として前提化**する（Square API 連携は v1 に送る）。販売モデルとしては **県の宿泊税関連の助成金制度（制度名・要件は今後確認）を財源とした、Mac / iPad / Square プリインストール済み端末キットの販売** を将来構想として志向する。紙台帳ベース運用は引き続き v0 のベースラインとして維持し、iPad 現場入力を採用しない事業者でも運用できる状態を崩さない。

**背景**:
- 沖縄北部の小規模宿泊事業者の端末保持率の低さが宿泊税導入の地域課題になっている
- ソフト単体を配布しても「そもそも動かす PC がない」事業者層にはリーチできない
- 「届いて電源を入れたら宿泊税運用が始められる」状態で提供することが普及の鍵
- 一方で v0 の第一顧客は自社民泊（ドッグフーディング）であり、事業者によっては既存 PC と紙運用で完結させたい層も残る
- アプリは Docker 運用でサーバ側 OS を選ばず、フロントもブラウザベースのため、技術的には OS を固定する必然性がない
- 決済は Square を前提としておけば、将来の API 連携（売上突合・入力省力化）の余地を残せる

**代替案**:
- A. ソフトのみ配布し、端末調達は事業者任せ
  - 端末保持率の低さが普及の壁になり、地域課題の解決に寄与しない
  - **採用しない**
- B. 端末キットを買い切り / サブスクで販売
  - 初期費用が新たな普及の壁になる
  - **採用しない**
- C. 端末キット + 県の助成金制度を活用した販売モデル
  - 事業者の初期費用負担を抑えつつ、キット配布で導入障壁を下げられる
  - **採用**
- D. アプリ動作要件を Mac 固定にする
  - Docker 運用のため技術的に OS を固定する必然性がない
  - Windows / Linux で既に PC を持っている事業者を排除してしまう
  - **採用しない**（推奨として Mac を掲げるに留める）

**影響**:
- `README.md` の「起点と方向性」「ロードマップ」に助成金キット販売構想を追記
- `docs/v0-requirements.md` に「動作前提」節を追加、Stay モデルに `payment_method` カラム追加、iPad 現場入力のユースケースを追記、オープンクエスチョンの「決済手段の扱い」を Square 前提で確定し削除
- `docs/tech-stack.md` にフロント節 iPad Safari 対応追記 + ハードウェア前提節新設
- `docs/non-functional-requirements.md` にタブレット運用時のセキュリティ最低ライン（画面ロック・紛失時対応）追記
- `docs/paper-ledger-template.md` は **変更なし**（紙台帳はベースライン運用として維持）
- `CLAUDE.md` のプロジェクト概要に前提デバイスと販売モデルを追記
- 助成金の具体的な制度名・要件・申請フローの確認は別タスク（v0 実装着手とは独立）
- Square API 連携のスコープ確定・POC は v1 課題
- **自社民泊でのドッグフーディング対象を拡張**: 従来は「v0 アプリの検証場」として自社民泊を位置付けていたが、本決定により **Mac / iPad / Square のキット構成そのもの** も同じ場で検証対象に含める。具体的には (1) キッティング手順が現実的か、(2) 受付〜決済〜入力〜月次申告の業務フローが iPad/Square 込みで回るか、(3) スタッフ教育に要する時間と躓きポイント、(4) 端末トラブル時のリカバリ運用、を自社運用の中で洗い出す。これらの知見が v1 の助成金キット販売モデル具体化の土台になる

## 2026-04-07: MonthlyClose モデルを v0 から外す

**決定**: v0 では月次確定（MonthlyClose）モデルを持たない。代わりに `MonthlyExport`（仮称、エクスポート実行時刻の記録）のみを持ち、Stay は常時編集可とする。エクスポート済月の編集時は警告表示で誤操作を防ぐ。事後監査は paper_trail に委ねる。

**背景**:
- 当初の v0 要件では「月次確定 → 編集ロック → 取消・訂正は履歴を残す」仕様を想定していた
- しかし「再オープンの権限・手順・再計算ポリシー」を詰めるコストが、得られる便益（誤操作防止）に対して釣り合わない
- v0 ユーザーは同一会社のスタッフ2〜3人、信頼モデルで運用される
- 条例第10条の申告納入は「毎月末日までに前月分」なので、法的な「確定」は県への申告納入の時点であってアプリ内状態ではない
- 編集ロックとロール権限は v1 で一緒に導入する方が、設計的にもまとまりがよい

**代替案**:
- A. MonthlyClose モデル + 再オープン権限管理 + 再計算ポリシー
  - v0 の「全員同権限」方針と衝突する（再オープン権限を誰に与えるか決められない）
  - 再計算を「当時の tax_rule_version で固定」するか「最新版で再計算」するかの判断も必要
  - **採用しない**: v0 スコープに対して過剰
- B. MonthlyExport（エクスポート実行時刻の記録）のみ + paper_trail で事後監査 + 編集警告
  - 最小実装で法令要件（帳簿・売上伝票の保存）は満たせる
  - 月次確定という概念そのものを v1 に送る
  - **採用**

**影響**:
- `docs/v0-requirements.md` §3.5 月次申告 / §4.4 月次締め を書き換え
- 「月次確定」「再オープン」「編集ロック」の記述を削除
- v1 のロードマップに「ロール権限 + 編集ロック」をセットで追加
- #12 で反映

## 2026-04-07: `config/facility.json` の概念を v0 では採用しない

**決定**: v0 段階では `config/facility.json` を作らない。施設固有設定は Rails credentials（テキスト設定）と `app/assets/images/facility/*`（バイナリ資産）に分けて管理する。Kamal デプロイ時は Kamal secrets で master.key を配布する。

**背景**:
- CLAUDE.md の設計原則「規律1: ユーザー固有データはコードに書かない → `config/facility.json` に集約」は理想像として正しい
- しかし v0 対象は自社1施設のみで、Kamal デプロイ時の受け渡し方法を決める負担が大きい
- 他社配布（他の宿泊事業者が deploy して使う）を見据えた汎用配布設計は v1 課題
- Rails credentials は Rails 標準機能で、追加の仕組み導入なしに秘密情報管理ができる
- 「コードに書かない」規律は credentials でも満たせる（credentials はコードではない、かつ暗号化されている）

**代替案**:
- A. 環境変数展開（`RAILS_FACILITY_JSON` に JSON 文字列を入れる）
  - 可読性が低い、バイナリ資産を扱えない
  - **採用しない**
- B. Volume マウント（ホスト側の `config/facility.json` をコンテナにマウント）
  - 他社配布時は有利だが、v0 段階では過剰
  - **採用しない**: v1 で再検討
- C. Rails credentials + assets 直置き
  - Rails 標準機能のみで完結、追加仕組み不要
  - テキスト設定とバイナリ資産を自然に分離できる
  - **採用**
- D. 初回セットアップ wizard で DB に保存
  - 運用者目線では最も楽だが、v0 実装負荷が高い
  - **採用しない**: v1 で再検討
- E. ビルド時埋め込み（deploy 用ブランチで commit）
  - 汎用化規律に反する
  - **採用しない**

**影響**:
- CLAUDE.md / README の「`config/facility.json` に集約」記述を修正
- テキスト設定（施設名・住所・登録番号・領収書但し書き等）→ `Rails.application.credentials.facility`
- ロゴ等バイナリ資産 → `app/assets/images/facility/*`、`.gitignore` 対象
- 「規律1」の文言を「ユーザー固有データはコードに書かない、Rails credentials または非コミット資産として管理する」に書き換え
- 汎用配布時の受け渡し方法の確定は v1 で改めて行う
- #12 で反映

## 2026-04-07: データモデルに `tax_rule_version` カラムを追加する

**決定**: Stay モデルに `tax_rule_version` カラムを追加する。v0 では実質単一値 `okinawa-2027-02-01` を固定で焼き付け、version 切替ロジックは v0 では実装しない。命名規則は「日付ベース + 自治体プレフィックス」、焼き付けタイミングは「Stay 入力時」とする。

**背景**:
- CLAUDE.md 規律2で「条例改定への対応は `tax_rule_version` フィールドで吸収し、過去の Stay は当時の version で再計算可能にする」と明記されている
- しかし `docs/v0-requirements.md` のデータモデルには `tax_rule_version` の記載がなく、設計メモと要件ドキュメントに乖離があった
- 沖縄県宿泊税条例附則7項「令和11年度を目途に見直し」により、条例改定は確実に発生する
- v0 運用開始（2027年2月）から約2年で改定イベントが来る想定
- 改定時に過去データを当時の条文で再計算できる必要がある（税務調査時の説明義務）

**代替案**:
- A. 追加しない（v0 では単一バージョンのみ運用、改定時に考える）
  - 改定時に過去 Stay の追跡不能、監査時の説明困難
  - マイグレーション後付けのコストが高い（Stay に NOT NULL カラム追加は本番運用後は厄介）
  - **採用しない**
- B. 追加するが version 切替ロジックも v0 で実装
  - v0 スコープに対して過剰（そもそも v0 期間中に複数バージョンは存在しない）
  - **採用しない**
- C. 追加するが v0 は単一値固定、切替ロジックは v1
  - 将来の改定コストを最小化しつつ、v0 の実装負荷は最小
  - **採用**

**命名規則の判断**: 日付ベース + 自治体プレフィックス（例: `okinawa-2027-02-01`）
- 条例附則の施行日と1:1対応し、監査時の説明が最短
- セマンティックバージョン（`v1` / `v2`）は改定内容の意味付けを我々が勝手にすることになり、税務判断を越えるため採用しない
- 自治体プレフィックスは将来の多自治体対応（v1以降の論点）への備え

**焼き付けタイミングの判断**: Stay 入力時に当時の current version を Stay に焼き付ける
- 月次確定時ではなく入力時に確定させることで、後日の current version 変更が過去データに影響しない
- paper_trail で変更履歴は別途残るため、誤った version で保存した場合の事後訂正は可能

**影響**:
- `docs/v0-requirements.md` の Stay データモデルに `tax_rule_version` カラム追加
- 税計算 Service Object（`OkinawaTaxCalculator`）のインターフェイスに version を引数として含める
- マイグレーションで NOT NULL カラムとして追加、初期値 `okinawa-2027-02-01`
- #12 で反映
